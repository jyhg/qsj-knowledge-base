# 数仓测试用例知识库与验数运营台 SPEC v2.0

## 1. 文档目的
基于 `docs/context.md` 与 `pics` 目录中的 3 张 DQC 平台截图，重新澄清本产品的目标、边界、对象模型与 MVP 范围。

本次澄清后的核心结论是：

1. 这不是一个新的 DQC 平台。
2. 这也不只是一个“AI 帮忙写 SQL”的工具。
3. 这是一个面向数仓开发与 PM 的“测试用例知识库 + one service 手动验数执行 + DQC 回填运营台”。
4. 产品的核心价值，不是把所有事情自动化，而是把分散在人脑、历史 SQL、历史事故、业务经验中的验数知识，沉淀成可检索、可执行、可解释、可回写、可通过 Git 管理的测试资产。

## 2. 产品重新定义
### 2.1 产品定位
产品可暂定名为：

`数仓测试用例知识库与验数运营台`

其定位不是“监控平台”，而是位于以下两者之间的中间层：

1. 上游：数仓开发、PM、分析场景、需求交付场景中的业务理解与人工验数经验。
2. 下游：现有 DQC 平台的规则模板、监控配置、定时调度、告警能力，以及数据平台 `one service` 的 SQL 接口执行能力。

本产品主要负责四件事：

1. 数仓/PM可以编辑维护并沉淀测试用例知识。
2. 在取数分析和开发自测时，筛出相关规则，并通过 `one service` 手动触发执行。
3. 输出验数结果、异常点和分析结论。
4. 在需求上线后，整理应发布到 DQC 的规则包，与 DQC 平台当前规则做差异比对，再由人工确认后更新。

### 2.2 与 DQC 平台、one service 的关系
结合截图与补充说明，可以确认：

1. 现有 DQC 平台只适合定时调度类监控。
2. 数据平台还提供 `one service`，可通过接口方式提交 SQL 并返回查询结果。

其中，现有 DQC 平台已经具备：

1. 规则模板能力。
2. 监控实例配置能力。
3. 调度频率与告警配置能力。
4. 已创建监控列表的管理能力。

因此本产品的责任边界应明确为：

1. 不替代 DQC 的定时调度、告警、实例管理。
2. 不重做 DQC 的监控配置页面。
3. 不自研 SQL 查询引擎，手动执行能力直接复用 `one service`。
4. 在取数分析和开发自测场景，通过 `one service` 承接“按表或按规则集合的即时手动验数”。
5. 在需求上线后的持续监控场景，通过 DQC 承接“长期、定时、可告警”的规则运行。
6. 本产品负责生成“应该验什么、为什么验、如何验、应该走 one service 还是 DQC”。
7. 本产品负责产出 `one service` 可执行的 SQL 任务清单，以及 DQC 可接收的规则草案、配置草案和差异清单。

### 2.3 产品本质
本产品本质上是“数仓测试用例知识库”。

这里的“测试用例”不是传统 QA UI 测试用例，而是针对数仓数据结果的：

1. 观测点定义。
2. 技术验数规则。
3. 业务判断知识。
4. `one service` 手动执行映射。
5. DQC 发布映射。

## 3. 为什么现在要做
当前团队面临的核心问题不是缺少某一条 SQL，而是缺少一套可持续积累的数据测试资产体系。

现状痛点包括：

1. 同一业务口径在多套代码里实现，结果存在差异。
2. 数仓开发在取数分析时，无法快速拿到所有相关验数规则。
3. 开发自测往往只能覆盖少量维度，难以覆盖关键维度组合。
4. 上线后进入 DQC 的规则不系统，常依赖开发者临时补。
5. 历史事故、业务常识、对账经验没有沉淀成结构化资产。
6. PM 与数仓都可能更新业务知识，但当前没有清晰的映射模型去承接。

所以需要的不是“多一个页面”，而是一个能把业务知识转换为测试用例资产的工具。

## 4. 目标与非目标
### 4.1 业务目标
1. 降低取数分析场景中的数据错漏、口径偏差与异常遗漏。
2. 降低开发自测阶段的漏检率，并提升手动验数的即时性。
3. 在需求上线后，提高 DQC 规则补齐的系统性与覆盖率。
4. 建立一套可长期维护的数仓测试用例知识体系。

### 4.2 产品目标
1. 以“表 + 维度组合下的值表现”为最小观测粒度沉淀知识。
2. 让用户能按需求涉及表，快速筛出相关测试用例，并通过 `one service` 手动执行。
3. 让每次执行结果都能产出结论：是否有错漏、异常点在哪里、依据是什么。
4. 让每次上线后都能生成“建议发布到 DQC 的规则清单”，并支持人工确认后更新。
5. 让业务知识、技术规则和 DQC 发布规则之间建立明确映射。
6. 让知识演进过程可基于 Git 进行版本管理、审阅、回滚和追溯。

### 4.3 非目标
1. 不自研新的 DQC 调度引擎。
2. 不在 MVP 阶段自动替用户决定哪条业务规则是真理。
3. 不在 MVP 阶段自动修改 DQC 线上规则并直接发布，必须人工确认。
4. 不在 MVP 阶段覆盖所有表和全部业务，只覆盖试点核心表。
5. 不做复杂组织权限系统，MVP 只区分数仓开发与 PM。
6. 不自研新的 SQL 执行引擎，手动执行统一复用 `one service`。

## 5. 核心设计判断
### 5.1 最小沉淀单元不是“表”也不是“规则”
如果直接把知识沉淀在“表”或者“规则”层，会遇到两个问题：

1. 粒度过粗，无法描述特定维度组合下的异常判断。
2. 无法表达一条业务知识同时作用于多个表、多个维度组合的情况。

因此最小沉淀单元应定义为：

`值表现观测点`

### 5.2 值表现观测点定义
一个值表现观测点描述的是：

`某张表在某个时间粒度、某组过滤条件、某组维度组合下，一个指标值应该如何被观察和比较`

建议结构为：

1. 表。
2. 指标。
3. 统计口径或聚合方式。
4. 时间粒度。
5. 维度组合。
6. 过滤条件。
7. 值来源 SQL 或逻辑模板。
8. 适用场景。

可简写为：

`Observation Point = Table + Metric + Time Grain + Dimensions + Filters + Aggregation`

### 5.3 四层模型
为了支撑你提到的复杂映射关系，本产品采用四层模型。

#### L1：值表现观测点 Observation Point
最小沉淀单元。

例子：

1. `ads_app_qsj_agg_cate_conv` 表，在 `日期 + 平台 + 玩法 + 经营类型` 维度下的 `转化率`。
2. `ads_app_qsj_zz_secondhand` 表，在 `日期 + 平台 + 渠道` 维度下的 `订单量`。

#### L2：技术测试用例 Technical Test Case
技术层的可执行验数规则。它消费 1 个或多个观测点，并定义比较方式、阈值、执行 SQL 模板、判定逻辑。

例子：

1. 同表同指标在相邻日期的波动不超过 30%。
2. 表 A 的某维度组合值应与表 B 的某维度组合值一致。
3. 某字段空值率不得超过 5%。

#### L3：业务规则 Business Rule
业务层的判断知识，用业务语言描述“什么是合理的、什么是不合理的”。

例子：

1. 某自媒体平台玩法下，自营转化应低于投流转化。
2. 某业务链路中，曝光 >= 点击 >= 转化。
3. 对账场景下，ADS 汇总值应与来源表汇总值一致。

#### L4：DQC 发布实例 DQC Deployment
将技术测试用例映射为 DQC 平台可接收的规则模板和监控实例配置。

至少包括：

1. DQC 规则类型。
2. 监控对象。
3. 判断字段。
4. 操作符与阈值。
5. 前置 SQL。
6. 调度频率。
7. 告警级别。
8. 接收人或接收群。

### 5.4 双执行通道
同一套测试资产需要支持两种执行通道：

1. `one service` 手动执行通道：
   1. 用于取数分析。
   2. 用于开发自测。
   3. 支持用户手动触发某个或某几个表的验数规则集合。
   4. 目标是即时返回 SQL 查询结果、异常点和分析结论。

2. DQC 定时执行通道：
   1. 用于需求上线后的长期监控。
   2. 支持定时调度和告警。
   3. 目标是把高价值、可长期复用的测试用例转成稳定监控项。

因此，测试用例本身不应只绑定 DQC，还应标记其执行通道属性：

1. 仅 `one service`
2. 仅 DQC
3. 同时支持两者

### 5.5 为什么必须做四层拆分
因为你给出的两个例子已经证明映射一定是多对多：

1. `ads 表1在某维度组合下的值 = ads 表2在某维度组合下的值`
这说明一条技术测试用例会引用多个观测点。

2. `某平台玩法下，自营转化 < 投流转化`
这说明一条业务规则会映射到多个观测点的对比关系。

3. 同一个观测点可能被多个业务规则复用。
比如一个 `转化率` 观测点，既会被“链路合理性”使用，也会被“渠道对比合理性”使用。

所以：

1. 业务规则不能直接绑死在单表单字段上。
2. 技术测试用例不能等同于业务规则。
3. DQC 发布实例也不能等同于知识本身。

10. 触发人
11. 触发时间

### 6.6 映射关系
必须支持以下关系：

1. 一个业务规则关联多个观测点。
2. 一个业务规则关联多个技术测试用例。
3. 一个观测点被多个业务规则复用。
4. 一个技术测试用例被部署为多个 DQC 发布实例。
5. 一个需求任务会命中多个观测点、多个测试用例和多个业务规则。

## 7. 目标用户
### 7.1 数仓开发
关注点：

1. 当前需求涉及哪些表。
2. 这些表有哪些历史测试用例。
3. 哪些用例可以直接执行。
4. 执行结果是否有异常、异常点在哪里。
5. 上线时哪些规则需要补进 DQC。

### 7.2 PM
关注点：

1. 某条业务知识是否已经被沉淀。
2. 当前表下的观测点、测试用例、业务规则是否需要补充或修订。
3. 某个业务规则对应了哪些技术测试用例。
4. 某类异常是否已有历史经验与分析结论。
5. 本次上线是否覆盖了关键业务风险。

### 7.3 团队负责人
关注点：

1. 核心表的测试用例覆盖率。
2. 哪些高风险表没有知识沉淀。
3. 哪些规则仍停留在人工经验层，没有转成可执行资产。

## 8. 典型使用场景
### 场景 A：取数分析前的人工验数
目标是先验证数据能不能拿、敢不敢给。

流程：

1. 用户输入需求背景、涉及表、涉及指标、时间范围。
2. 系统按表召回相关观测点、测试用例、业务规则。
3. 系统输出“推荐执行清单”，用户手动勾选并触发。
4. 系统将命中的 SQL 通过 `one service` 接口方式提交到数据平台。
5. 系统接收 `one service` 返回结果并汇总。
6. 系统汇总结论：
   1. 是否发现数据错漏。
   2. 是否发现异常点。
   3. 哪些异常是技术异常。
   4. 哪些异常更可能是业务违背常识。
   5. 结论依据了哪些规则和知识。

### 场景 B：开发自测
目标是在开发提交前把相关测试用例跑一遍。

流程：

1. 用户输入需求、改动 SQL 或改动表。
2. 系统按受影响表召回历史测试用例。
3. 系统将测试用例分为：
   1. 基础检查。
   2. 业务关键检查。
   3. 上线后建议沉淀到 DQC 的检查。
4. 用户通过 `one service` 手动执行自测规则集合。
5. 系统输出结果与分析结论，并允许补充新知识。

### 场景 C：需求上线后回填 DQC
目标不是重新验数，而是把应长期运行的规则补入 DQC。即时、临时、自测型规则继续走 `one service`，不强行进入 DQC。

流程：

1. 用户选择本次需求涉及表。
2. 系统召回这些表对应的“可发布到 DQC”的测试用例。
3. 系统拉取或录入当前 DQC 平台已存在规则。
4. 系统输出差异清单：
   1. 已有规则。
   2. 建议新增规则。
   3. 建议更新规则。
   4. 建议下线规则。
5. 用户人工确认。
6. 系统生成 DQC 更新包并执行同步，或导出待录入清单。

### 场景 D：知识维护
目标是让数仓开发与 PM 能持续沉淀测试资产。

流程：

1. 用户新增或编辑业务规则、观测点、测试用例。
2. 系统要求填写结构化字段，而不是只写备注。
3. 系统要求将变化提交到 Git 版本管理。
4. 审核通过后，知识状态从草稿变为生效。

## 9. 产品原则
1. 观测点先于规则。
2. 技术测试用例与业务规则分层管理。
3. AI 只负责辅助生成与召回，不替代最终判断。
4. 任何关键动作都必须有人确认。
5. 任何结论都必须能追溯到规则、观测点和执行证据。
6. 即时手动执行与长期定时监控必须分通道设计：`one service` 负责前者，DQC 负责后者。
7. 任何长期有效的规则都要考虑是否值得回填 DQC。
8. 知识资产必须支持 Git 化管理，而不是只保存在系统数据库里。

## 10. MVP 范围
### 10.1 MVP 必须跑通的闭环
MVP 至少要跑通以下三条闭环：

1. `需求涉及表 -> 召回测试用例 -> 通过 one service 手动执行 SQL -> 输出异常结论`
2. `需求上线 -> 选中涉及表 -> 生成 DQC 差异清单 -> 人工确认 -> 更新 DQC`
3. `执行中发现新规则 -> 结构化沉淀 -> Git 版本化保存`

### 10.2 MVP 首批覆盖对象
建议：

1. 只选择 3 到 5 张高风险 ADS 表。
2. 每张表先沉淀 5 到 10 个核心观测点。
3. 每张表先沉淀 5 到 10 个高价值测试用例。
4. 至少覆盖 2 类业务规则：
   1. 表间一致性类。
   2. 平台 / 玩法 / 经营模式之间的业务约束类。

### 10.3 MVP 不必一步到位的内容
1. 不必先做自动推荐全部维度组合。
2. 不必先做复杂审批流。
3. 不必先做自动发布 DQC，可先做“差异清单 + 导出 / 半自动同步”。

## 11. 功能需求
### FR1：测试资产管理
系统必须能管理以下四类对象：

1. 观测点。
2. 技术测试用例。
3. 业务规则。
4. DQC 发布实例。

### FR2：按表筛选相关测试用例
在取数分析和开发自测场景中，系统必须支持：

1. 以表为主要入口筛选。
2. 补充按指标、时间粒度、维度组合筛选。
3. 展示该表关联的全部测试用例和业务规则。

### FR3：手动触发执行
系统必须支持：

1. 用户勾选一个或多个测试用例，支持按某个或某几个表批量勾选规则集合。
2. 用户手动触发执行。
3. 执行时必须通过 `one service` 接口调用只读 SQL 能力。
4. 系统必须接收并解析 `one service` 返回结果。
5. 对每个测试用例返回结构化结果。

### FR4：分析结论输出
每次执行后，系统必须至少输出：

1. 是否通过。
2. 是否疑似数据错漏。
3. 是否存在异常点。
4. 异常点落在哪些维度组合。
5. 引用了哪些业务规则和技术测试用例。
6. `one service` 返回的核心结果摘要。
7. 建议人工关注什么。

### FR5：业务规则与测试用例映射
系统必须允许：

1. 一条业务规则绑定多个测试用例。
2. 一条测试用例绑定多个观测点。
3. 一个观测点被多个业务规则复用。

### FR6：DQC 差异比对
系统必须支持：

1. 选择需求涉及表。
2. 召回这些表应发布的测试用例。
3. 导入或读取当前 DQC 规则。
4. 输出新增 / 更新 / 下线 / 无变化的差异结果。

### FR7：DQC 发布映射
系统必须能把测试用例映射到 DQC 的配置字段，至少覆盖截图中体现的能力：

1. 规则模板类型。
2. 监控对象。
3. 判断字段。
4. 前置 SQL。
5. 操作符与阈值。
6. 执行周期。
7. 告警等级。
8. 告警接收人。

### FR8：one service 批次执行与结果留痕
系统必须支持：

1. 将一次手动验数视为一个批次执行。
2. 一个批次可包含多个表、多个测试用例、多个 SQL。
3. 保留 `one service` 请求信息、返回结果摘要和执行状态。
4. 支持按批次回看执行结论。

### FR9：知识补充与维护
系统必须支持数仓开发和 PM：

1. 新增业务规则。
2. 新增观测点。
3. 新增测试用例。
4. 建立映射关系。
5. 废弃旧规则。
6. 记录补充原因与适用场景。

### FR10：知识不足显式暴露
当系统找不到合适知识时，必须提示：

1. 当前缺的是观测点。
2. 当前缺的是技术测试用例。
3. 当前缺的是业务规则。
4. 当前无法直接同步到 DQC。

### FR11：审计与追溯
系统必须保留：

1. 执行人。
2. 执行时间。
3. 执行 SQL 摘要。
4. 执行结果。
5. 结论。
6. 关联知识版本。
7. `one service` 请求 / 返回摘要。
8. DQC 同步前后的差异记录。

### FR12：角色权限
MVP 角色只分两类：

1. 数仓开发：
   1. 可执行 SQL。
   2. 可确认 DQC 更新。
   3. 可提交知识变更。
   4. 可新增、编辑、删除观测点、测试用例、业务规则。
2. PM：
   1. 可查看结果。
   2. 可参与人工确认。
   3. 可提交知识变更。
   4. 可新增、编辑、删除观测点、测试用例、业务规则。

说明：知识资产维护权限在 MVP 中对 PM 与数仓开发完全同权；若执行 `one service`、DQC 同步、回滚等动作仍有差异，需单独建模，不再与知识维护能力混写。

### FR13：Git 版本管理
知识资产的变更必须基于 Git 做版本管理。

至少满足：

1. 每次知识变更都有变更集。
2. 每次变更都有提交人、时间、原因。
3. 支持按对象查看历史版本。
4. 支持回滚到历史版本。
5. 支持对比两个版本的差异。

## 12. Git 版本管理方案
### 12.1 为什么必须基于 Git
因为这类知识不是一次性配置，而是持续演进的测试资产。

Git 的价值在于：

1. 可审阅。
2. 可回滚。
3. 可追责。
4. 可关联需求。
5. 可与开发流程统一。

### 12.2 Git 管理对象
建议将以下对象结构化落盘并纳入 Git：

1. 观测点定义文件。
2. 技术测试用例定义文件。
3. 业务规则定义文件。
4. DQC 映射定义文件。
5. 同步到 DQC 的规则包快照。

### 12.3 建议目录
建议目录如下：

```text
knowledge-assets/
  observation-points/
  test-cases/
  business-rules/
  dqc-mappings/
  releases/
```

### 12.4 变更流程
建议流程：

1. 用户在产品中编辑知识。
2. 系统生成结构化文件变更。
3. 系统要求填写变更原因、适用场景、关联需求。
4. 系统生成 Git 变更集。
5. 人工确认后提交。
6. 生效版本写回系统索引。

### 12.5 Git 与数据库的关系
建议原则：

1. Git 是知识资产的权威版本源。
2. 数据库是运行态索引、任务、`one service` 执行记录、DQC 同步记录和缓存载体。
3. 运行记录不强制全部进 Git。
4. 知识定义与 DQC 发布快照必须进 Git。

## 13. 关键页面与操作台能力
### 13.1 页面一：测试用例知识库
至少支持：

1. 按表查看。
2. 查看该表的观测点。
3. 查看关联测试用例。
4. 查看关联业务规则。
5. 查看是否已映射到 DQC。

### 13.2 页面二：验数执行台
至少支持：

1. 输入需求上下文。
2. 选择涉及表。
3. 展示相关测试用例。
4. 通过 `one service` 手动触发执行。
5. 展示 `one service` 返回结果、异常点和分析结论。

### 13.3 页面三：DQC 差异回填台
至少支持：

1. 选择需求涉及表。
2. 生成建议发布清单。
3. 对比当前 DQC 已有规则。
4. 勾选确认新增 / 更新 / 下线。
5. 触发同步或导出结果。

### 13.4 页面四：知识变更记录
至少支持：

1. 查看版本历史。
2. 查看 Git 提交信息。
3. 查看差异内容。
4. 触发回滚。

## 14. 输出物定义
### 14.1 验数执行结果
必须包含：

1. 执行的测试用例列表。
2. 每条测试用例的结果。
3. 涉及的维度组合。
4. 异常点。
5. 分析结论。
6. 引用的业务规则。
7. `one service` 返回摘要。

### 14.2 异常分析结论
必须回答三个问题：

1. 是否有数据错漏。
2. 异常点在哪里。
3. 为什么判断它异常。

### 14.3 DQC 差异清单
必须包含：

1. 已存在规则。
2. 建议新增规则。
3. 建议更新规则。
4. 建议下线规则。
5. 每条规则的来源测试用例和业务依据。

### 14.4 Git 变更记录
必须包含：

1. 变更对象。
2. 变更原因。
3. 关联需求。
4. 提交人。
5. 提交时间。
6. 版本号。

## 15. 验收标准
### 15.1 MVP 验收
满足以下条件才算 MVP 可用：

1. 至少 3 张试点表建立了观测点、测试用例、业务规则三层资产。
2. 用户能按涉及表筛出相关测试用例。
3. 用户能按一个或多个表，通过 `one service` 手动执行 SQL 验数。
4. 系统能输出异常点和分析结论。
5. 系统能对需求涉及表生成 DQC 差异清单。
6. 用户能人工确认后更新 DQC。
7. 用户能在系统中维护知识，并形成 Git 版本记录。

### 15.2 成功标准
如果以下结果出现，可认定方向成立：

1. 取数分析前的准备时间明显下降。
2. 开发自测覆盖率明显提升。
3. 上线后补 DQC 的规则更系统。
4. 历史验数知识不再主要依赖个人经验。

## 16. 风险与应对
1. 风险：业务规则难以结构化。
应对：先拆成“观测点 + 技术测试用例 + 业务规则”三层，不直接追求一层表达全部语义。

2. 风险：AI 生成的规则不可靠。
应对：AI 只生成草案，最终以人工确认和 Git 入库为准。

3. 风险：DQC 映射类型过多，首版做不全。
应对：先覆盖截图中的主流模板类型，如缺失、唯一、一致、波动、大小关系。

4. 风险：知识维护成本高。
应对：先聚焦高风险核心表，只维护高价值用例。

5. 风险：`one service` 返回结果过大、超时或解析不稳定。
应对：限定手动执行批次大小；保留请求 ID；支持结果采样、超时提示和失败重试。

6. 风险：产品和 Git 流程脱节。
应对：知识变更必须要求填写原因并生成版本记录，不允许只在数据库里改。

## 17. 当前明确结论
截至本轮，已可以明确：

1. 产品目标应从“数据质量知识库”进一步收敛为“数仓测试用例知识库”。
2. 最小粒度应是“表 + 维度组合下的值表现观测点”。
3. 业务规则、技术测试用例、DQC 发布实例必须分层建模。
4. 产品要同时支持两个核心场景：
   1. 取数分析 / 开发自测时筛规则，并通过 `one service` 手动执行。
   2. 需求上线后按涉及表回填 DQC。
5. 一条业务规则对应多个表 + 维度组合是正常情况，系统必须原生支持多对多映射。
6. 知识维护主体可以是数仓开发和 PM，但版本治理必须走 Git。
7. 执行通道必须双轨：`one service` 负责即时手动验数，DQC 负责上线后的定时监控。
8. MVP 的重点不是“自动”，而是“沉淀、筛选、执行、解释、回填”。

## 18. 苏格拉底下一轮待澄清问题
以下问题不阻塞本版 SPEC，但会直接影响下一轮产品细化，需要在后续对话中继续收敛：

1. 观测点中的“维度组合”是否需要区分：
   1. 展示维度。
   2. 过滤维度。
   3. 对比维度。

2. 一条业务规则是否要支持“适用条件”与“例外条件”？
   例如某平台玩法下一般成立，但某些活动日不成立。

3. DQC 回填时，首版是否只做“导出待配置清单”，还是要直连 DQC API 做半自动同步？

4. PM 与数仓开发在知识资产维护上完全同权：两者都允许维护观测点、测试用例和业务规则；权限差异仅在执行、发布、回滚等动作上另行定义。

5. 观测点是否需要明确区分：
   1. 来源表观测点。
   2. 结果表观测点。
   3. 对账桥接观测点。

6. 首批试点的 3 到 5 张表分别是谁？
   如果不明确试点对象，MVP 很容易继续抽象化而不落地。

---

本版 SPEC 的核心产出，不是把所有细节都定死，而是先把产品抽象从“规则库”校正为“测试用例知识库”，并建立起 `观测点 -> 测试用例 -> one service 手动执行 / DQC 发布 -> 业务规则解释` 的正确运行闭环。

# 数仓测试用例知识库与验数运营台 页面原型与信息架构 v2.0

基于 [SPEC.md](/Users/zz/Documents/gitproject/qsj-knowledge-base/docs/SPEC.md#L1) 输出。本文件用于将产品需求翻译为页面结构、站点地图、关键页面职责与低保真原型。

本版已根据对话结论收敛两个关键判断：

1. 知识库以“表”为第一入口，而不是以“观测点 / 测试用例 / 业务规则 / DQC 发布实例”四类对象为第一入口。
2. 表详情页默认优先展示“测试用例”，因为用户进入某张表后最常见的动作是先看“这张表有哪些可以立即执行的验数用例”。

## 1. 文档目标
本文件回答 5 个问题：

1. MVP 阶段站点到底有哪些页面。
2. 页面如何围绕“表优先”展开。
3. 用户如何从“选择表”走到“手动执行 one service 验数”。
4. 用户如何从“需求涉及表”走到“DQC 差异回填”。
5. 页面如何承接 `观测点 / 测试用例 / 业务规则 / DQC 发布实例 / Git 版本` 这些核心对象。

## 2. 设计原则
页面设计遵循以下原则：

1. 表优先，不做对象优先。
2. 执行动作优先，不做纯资料库首页。
3. 手动即时验数与长期定时监控分通道展示：`one service` 与 DQC 必须明确分开。
4. 测试用例优先于观测点展示，观测点优先于业务规则展示。
5. 异常事实优先，不先展示复杂推理链。
6. 知识沉淀必须结构化，不允许只写一段备注。
7. 版本与执行记录必须可追溯，Git 版本信息要有落点。

## 5. 角色视图
### 5.1 数仓开发
主要动作：

1. 按表浏览资产。
2. 在表详情中选择测试用例。
3. 发起 `one service` 手动验数。
4. 查看异常结论。
5. 确认回写知识。
6. 确认 DQC 回填。
7. 查看 Git 变更和回滚。

### 5.2 PM
主要动作：

1. 按表查看观测点、测试用例、业务规则。
2. 补充或编辑业务规则草稿。
3. 查看执行结果和异常解释。
4. 参与确认哪些规则应进入 DQC。
5. 查看版本变更。

限制：

1. 不触发 `one service` 执行。
2. 不确认最终 DQC 更新。
3. 不执行回滚。

## 7. 关键页面原型
## 7.1 工作台
### 页面目标
作为首页，帮助用户快速进入三类主流程，并看到与自己有关的待办与风险。

### 核心模块
1. 三个主入口卡片：`分析验数`、`开发自测`、`DQC 回填`。
2. 我的待办：待确认回写、待确认 DQC 更新、结果失效提示。
3. 最近执行批次。
4. 高风险表快照。

### 低保真原型
```text
+---------------------------------------------------------------------------------------------------+
| 标题：数仓测试用例知识库与验数运营台                                                               |
+---------------------------------------------------------------------------------------------------+
| [分析验数]                  [开发自测]                  [DQC 回填]                                |
| 创建 one service 执行任务    创建自测任务               创建回填任务                              |
+---------------------------------------------------------------------------------------------------+
| 我的待办                                                                                          |
| - 待确认回写 2 个   - 待确认 DQC 更新 1 个   - 结果失效 3 个                                      |
+---------------------------------------------------------------------------------------------------+
| 最近执行批次                                                                                      |
| 批次名称 | 场景 | 触发表 | 状态 | 异常数 | 最近执行时间                                           |
+---------------------------------------------------------------------------------------------------+
| 高风险表快照                                                                                      |
| 表名 | 测试用例数 | 最近异常 | DQC 覆盖情况 | 入口                                                |
+---------------------------------------------------------------------------------------------------+
```

```text
+---------------------------------------------------------------------------------------------------+
```

## 8. 关键交互流

```text
工作台 -> 创建分析验数任务 -> 规则集合选择 -> 通过 one service 执行
      -> 执行结果 -> 批量回写 -> 已完成
```

## 8.2 开发自测主流程
```text
工作台 -> 创建开发自测任务 -> 规则集合选择 -> 通过 one service 执行
      -> 执行结果 -> 批量回写 -> 已完成
```


```text

```

```text

```

## 9. 页面状态模型
### 9.1 任务状态
1. 草稿。
2. 待选择规则集合。
3. 执行中。
4. 待确认回写。
5. 待确认 DQC 更新。
6. 已完成。

### 9.2 表资产状态
1. 草稿。
2. 生效。
3. 废弃。

### 9.3 空状态
每个核心页面都需要空状态文案：

1. 表列表无命中结果。
2. 当前表暂无测试用例。
3. 当前表暂无观测点。
4. 当前表暂无业务规则。
5. 当前表暂无 DQC 映射。
6. 当前用户无待处理提示。

### 9.4 异常状态
需要单独设计错误状态：

1. `one service` 请求失败。
2. `one service` 返回超时。
3. DQC 差异获取失败。
4. 知识不足，无法生成可执行规则集合。
5. Git 版本写入失败。
